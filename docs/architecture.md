# Архитектура системы

## Обзор

Маркетплейс построен по принципу микросервисной архитектуры:

- **python-backend** (port 8000) — основной API. Управляет пользователями, каталогом услуг и жизненным циклом заказов. При платёжных операциях делает синхронные HTTP-вызовы в go-payment.

- **go-payment** (port 8001) — изолированный платёжный микросервис. Хранит состояние эскроу-счетов и ведёт блокчейн-лог транзакций. Написан на Go для высокой производительности и типобезопасности.

- **PostgreSQL** — единая СУБД. Схема `public` используется Python-бэкендом (через asyncpg + SQLAlchemy), схема `payments` зарезервирована для Go-сервиса.

- **Redis** — кэш JWT-токенов и временное хранилище сессионных данных.

## Блокчейн-лог

Каждая операция (lock/release/refund) порождает запись `Transaction`:

```
hash = SHA256(id + timestamp + from + to + amount + prev_hash)
```

Первая транзакция: `prev_hash = "0000000000000000"`. Каждая последующая содержит хэш предыдущей — это делает любое изменение исторической записи обнаруживаемым. Публичный эндпоинт `GET /chain/verify` пересчитывает все хэши и проверяет цепочку linkage.

## Принятые архитектурные решения

| Решение | Причина |
|---------|---------|
| Два сервиса вместо монолита | Изоляция платёжной логики, независимое масштабирование |
| In-memory store в Go | Упрощает прототип; в production заменяется на PostgreSQL-схему payments |
| asyncpg + SQLAlchemy async | Неблокирующий I/O для FastAPI |
| JWT Bearer (без cookies) | Совместимость с мобильными клиентами и API-потребителями |
| bcrypt для паролей | Индустриальный стандарт, устойчив к rainbow-table атакам |
